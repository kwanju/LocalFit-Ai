# ADR-023: 컨디션 트래킹 — 자가보고 체크인 (외부 건강 API 없음)

- **상태**: Proposed (2026-06-10)
- **관련 ADR**: ADR-024 (플랜 강도 조절), ADR-025 (메모리), ADR-028 (체력검증), ADR-002 (로컬-only)

## 컨텍스트

v4 비전의 핵심 차별점은 **컨디션 기반 운동 강도 조절**이다. 컨디션을 어떻게 얻을지가 문제 — 자가보고 vs 외부 건강 API(Apple Health / Google Fit) 연동.

deep interview(§6-4)에서 사용자는 **자가보고 체크인**을 선택했다. 외부 건강 API 는 플랫폼 의존(iOS/Android·기기)이 크고 우선순위가 낮다고 판단. 로컬-only(ADR-002) 유지에도 부합.

## 결정

- **자가보고 체크인**: 세션 시작 전(또는 일일) 간단 입력 — 피로도·근육통 등 1~5 척도 + 선택적 자유 메모.
- **외부 건강 API 미연동** (Apple Health / Google Fit) — 로컬-only 유지. 후순위(필요 시 별도 ADR).
- **체크인은 선택**(강제 X). 건너뛰면 기본 강도로 진행.
- 컨디션은 **플랜(ADR-024) 강도 조절의 입력**. 조절 자체는 코치가 제안 → 사용자 동의(ADR-024) — 컨디션이 자동으로 강도를 바꾸지 않는다.
- 저장: SQLite `condition_checkins`(date, fatigue, soreness, note). 자유 메모는 메모리(ADR-025) 2층과 연계 가능.

## 결과

### 긍정
- 외부 의존 0 — 로컬-only, 플랫폼 무관
- 가장 단순하게 컨디션 신호 확보
- 선택형이라 사용 부담 낮음

### 부정
- 자가보고는 주관적·부정확 가능 (보수적 강도 + 빠른 재조정으로 흡수)
- 수면·걸음수 같은 객관 데이터 부재
- 체크인 미입력 시 컨디션 신호 없음(기본 강도 fallback)

## 대안

| 후보 | 탈락 사유 |
|---|---|
| Apple Health / Google Fit 연동 | 플랫폼 의존 + 우선순위 낮음 — 후순위로 보류 |
| 컨디션 자동 강도 조절 | 회고 "LLM 마음대로" 경고 — 조절은 ADR-024 확인 게이트 |
| 체크인 강제 | 사용 마찰 — 선택형이 채택률 높음 |
