# ADR-006: TTS — Qwen3-TTS + SDPA 어텐션 + sentence-streaming

- **상태**: Accepted (2026-05-31)
- **관련 ADR**: ADR-002 (로컬-only), ADR-011 (Pipecat), ADR-012 (도메인 어댑터), ADR-013 (능동 코치 지연 의존성)
- **v1 이력 보존**: `_archive/v1/006-tts.md` — Kokoro → Qwen3-TTS → XTTS v2 3차 개정 회고
- **2026-06-02 개정**: phase-3 진행 중 upstream(`qwen-tts` PyPI 패키지)의 실제 API가 가정과 달라 본문 일부 정정. ① 로딩 클래스 `qwen_tts.Qwen3TTSModel` (transformers `AutoModel` 아님), ② public token-level streaming API 부재 → **sentence-batch streaming** 채택, ③ MeloTTS 병행 어댑터 추가 (config 토글)
- **2026-06-02 추가 개정 (실측 후)**: 5x 첫 청크 측정 결과 **Qwen3=4818ms / Melo=140ms**. Qwen3는 PRD §4-1 500ms 목표의 10배 초과로 실용 불가. **기본 active=`melo`로 전환**, Qwen3는 보이스 클로닝 옵션으로만 유지 (외부 ref WAV가 필요할 때 사용자가 수동 토글)
- **2026-06-07 개정 (백엔드 교체)**: Qwen3 백엔드를 `qwen-tts` → **`faster-qwen3-tts`**(CUDA Graphs 가속, 동일 `Qwen3-TTS-12Hz-1.7B-Base` 가중치)로 교체. 5x 첫 청크 **4818ms → 884ms**(RTX 5090, sentence-batch, ~5.5x 개선). 여전히 500ms 예산은 미달이나 PRD §4-1 (C2C 0.8~1.5s) 충족 — **기본 active=`qwen3`로 복귀**, Melo는 롤백 토글로 유지. 상세 §faster-qwen3-tts 백엔드. token-level streaming(`generate_voice_clone_streaming`) 도입 시 첫 청크 ~390ms 가능 — 필요 시 후속.

## 컨텍스트

### v1·v2 시점의 실패 회고

| 시점 | 결정 | 결과 |
|---|---|---|
| v1 초안 | Kokoro 단독 | 한국어 미지원으로 폐기 |
| v1 1차 | Kokoro + Qwen3-TTS 교체형 | Kokoro 한국어 fork 부재로 무의미 |
| v1 2차 | Qwen3-TTS 단독 | RTX 5090에서 합성 자체는 동작 |
| v1 Phase 6 | `config.tts.active = "qwen3"` 출시 | 사용자 청취 시 합성 지연 **~50초** 측정 |
| v2 시점 | XTTS v2 전환 (CPML 라이선스 수용) | 한국어 품질 미달 + Windows 빌드 까다로움 + 라이선스 외부 배포 불가 |

### 진짜 원인 — flash-attn 부재 + SDPA 미설정

v3-rewrite 사전 조사에서 결정적 사실 확인:

- **Qwen3-TTS는 Dual-Track hybrid streaming 아키텍처를 가지며 end-to-end 합성 지연이 97ms까지 가능** (공식 모델 카드).
- 단, **flash-attn 2가 공식 권장이지만 Windows에서 빌드 불가**.
- 커뮤니티 fork는 PyTorch 내장 `SDPA(Scaled Dot Product Attention)`로 대체 (`--no-flash-attn`) — Windows 10/11에서 정상 동작.
- v1 reference 스크립트가 `attn_implementation=None`(no SDPA) + `device_map=None`(CPU 폴백) 기본값을 사용 → 50초 지연 발생.
- 즉 **v1 50초 지연은 Qwen3-TTS 자체 한계가 아니라 잘못된 설정의 결과**다. CPML 라이선스 부담을 안고 XTTS로 갈 이유가 없다.

### Pipecat 통합

Pipecat은 Qwen3-TTS 공식 어댑터를 제공하지 않는다. `pipecat.services.tts.base.TTSService`를 상속해 자체 어댑터를 작성한다(ADR-012). 그러나 어댑터는 얇은 래퍼(transformers.AutoModel + inference) — Pipecat의 frame 처리·인터럽트·sentence aggregation은 부모 클래스가 제공한다.

## 결정

### 기본 TTS = MeloTTS (MIT) — 2026-06-02 실측 후 전환

> **개정 사유**: Phase 3 실측 5x 첫 청크 측정 결과 — Qwen3=4818ms, Melo=140ms. Qwen3는 sentence 한 개 동기 합성 비용이 ~5초로 PRD §4-1 (S2S 2~3s, C2C 0.8~1.5s) 충족 불가. MeloTTS는 한국어 자연성·지연 모두 합격 (사용자 청취 검증은 잔여 항목으로 분리).

- **모델**: MeloTTS Korean (`myshell-ai/MeloTTS-Korean`, HF 자동 다운로드)
- **로딩**: `from melo.api import TTS; TTS(language='KR', device='cuda:0')`
- **인터페이스**: `tts_to_file(text=..., speaker_id=0, output_path=None, speed=1.0)` — output_path=None이면 numpy waveform 직접 반환
- **샘플레이트**: 44.1 kHz mono (Pipecat `TTSAudioRawFrame.sample_rate=44100`으로 전송, transport가 필요 시 resample)
- **보이스**: preset speaker 0 (한국어는 단일 화자). 보이스 클로닝 필요 시 사용자가 config.tts.active를 "qwen3"로 토글
- **라이선스**: MIT — 상업 OK
- **G2P 백엔드 (Windows)**: g2pkk는 Windows에서 `eunjeon`(MSVC 빌드 필요)을 기본으로 시도하므로, `python-mecab-ko` (Windows 사전 빌드 wheel)를 사용하도록 `MeloTTSClient._patch_g2pkk_for_windows()`에서 monkey-patch. 또한 `melo.text.japanese`가 import 시점에 `MeCab.Tagger()`를 호출하므로 한국어 전용 사용 환경에서는 `MeCab` 모듈을 stub으로 대체 (`_install_japanese_mecab_stub`)
- **첫 청크 지연**: 워밍업 후 5x 평균 140ms (RTX 5090 + cu128)

### 보조 TTS = Qwen3-TTS (Apache 2.0) — voice clone 옵션

- **모델**: `Qwen/Qwen3-TTS-12Hz-1.7B-Base` — **Base 채택 사유 = 보이스 클로닝 zero-shot 지원**. non-Base 변형은 프리셋 화자만 지원하므로 채택하지 않음
- **로딩 클래스**: `qwen_tts.Qwen3TTSModel.from_pretrained(...)` — upstream이 별도 PyPI 패키지(`qwen-tts`)로 배포. transformers `AutoModel`/`AutoProcessor`로는 로드되지 않음 (2026-06-02 정정)
- **라이선스**: Apache 2.0 — 상업·외부 배포 모두 OK (XTTS CPML 부담 없음)
- **어텐션 구현**: `attn_implementation="sdpa"` 우선 시도, 미지원 시 `"eager"` 폴백 — PyTorch 내장 SDPA 사용 (flash-attn 불필요)
- **디바이스 매핑**: `device_map="cuda:0"` (또는 `cuda` — 모든 텐서 GPU 강제)
- **dtype**: `torch.bfloat16` (RTX 5090 + Ampere/Ada/Blackwell 지원)
- **샘플레이트**: 24kHz mono (Qwen3-TTS 출력 그대로)

### faster-qwen3-tts 백엔드 (2026-06-07 채택 — 기본 TTS로 복귀)

> **개정 사유**: `qwen-tts` 백엔드는 호출당 ~5초 고정 비용으로 실용 불가였다. [`faster-qwen3-tts`](https://github.com/andimarafioti/faster-qwen3-tts)는 **동일 `Qwen3-TTS-12Hz-1.7B-Base` 가중치**를 쓰면서 predictor/talker decode 단계를 CUDA Graphs로 캡처해 재생한다. RTX 5090 실측 sentence-batch 첫 청크 **884ms** (5x 평균), HOT RTF ~0.34x. 5x 개선으로 PRD §4-1 (C2C 0.8~1.5s) 충족 → **기본 active=`qwen3` 복귀**, Melo는 config 토글 롤백 경로로 유지.

- **로딩 클래스**: `faster_qwen3_tts.FasterQwen3TTS.from_pretrained(model_id, device="cuda:0", dtype=torch.bfloat16, attn_implementation="sdpa")` — `device`/`dtype`/`attn_implementation` 인자 그대로 수용. sdpa→eager 폴백 유지.
- **보이스 클로닝 캐싱**: `FasterQwen3TTS`가 `(ref_audio, ref_text, ...)` 키로 prompt를 **내부 캐싱**. 매 호출 `ref_audio`/`ref_text`만 전달 — 첫 호출만 추출, 이후 재사용. (`qwen-tts`의 `create_voice_clone_prompt` 수동 캐싱은 faster에 없음 → 불필요.)
- **warmup**: 첫 `generate_voice_clone` 호출에서 CUDA Graph 캡처(~6.5s)가 lazy 발생하므로 `Qwen3TTSClient.__init__`에서 throwaway 합성 1회로 startup 시점에 흡수 — 사용자 첫 발화는 ~470~900ms.
- **streaming**: `generate_voice_clone_streaming(..., chunk_size=12)`로 token-level streaming 가능 (첫 청크 ~390ms). 현재는 sentence-batch 유지(Option A), 더 낮은 TTFA 필요 시 도입(Option B).
- **롤백**: faster 백엔드 이슈(예: Blackwell CUDA Graph) 시 `tts.active: "melo"`로 즉시 전환. `qwen-tts` 패키지는 안정화 확인 후 제거.

### Sentence-batch streaming (지연 단축 핵심)

> **2026-06-02 정정**: 초안은 transformers `generate_stream`-style token streaming을 가정했으나 upstream `qwen_tts.Qwen3TTSModel`은 public token-level streaming API가 없다 (모든 메서드가 `Tuple[List[np.ndarray], int]` 반환). 따라서 **sentence-batch streaming** 채택 — 한 sentence는 동기 합성하되 sentence 단위로 즉시 재생 시작.

PRD v4 §4-1 목표(C2C 0.8~1.5s, S2S 2~3s) 충족 전략:

1. **LLM 응답을 sentence 단위로 절단** → 첫 sentence 합성 즉시 재생 시작, 다음 sentence는 백그라운드 합성.
2. **Pipecat의 `SentenceAggregator` + custom `Qwen3TTSService.run_tts(text)` 결합** — sentence boundary마다 frame 흘려보냄.
3. 평균 4문장 응답 기준 첫 청크 지연 < 500ms 목표. "첫 청크"의 측정 단위는 **첫 sentence 완전 합성 완료 시점** (token-level streaming이 아니므로 한 sentence의 부분 출력은 없음).
4. MeloTTS도 동일 패턴 사용 — Melo는 내부 `split_sentences_into_pieces` + per-sentence `model.infer()` 루프를 fork하여 동일 인터페이스 제공.

### 음성 클로닝 정책

- `ref_audio_path` 1회 캐싱 (`get_conditioning_latents` 결과를 `(speaker_embedding, conditioning_latent)` tuple로 메모리 상주)
- 첫 합성 후 latent 재사용 — 합성 외 오버헤드 0
- ref WAV는 6~10초 한국어 단일 화자 (`data/ref_voice.wav`). 외부 배포 전환 시 라이선스 클린한 ref로 교체.

### 지연 폴백 정책

Qwen3·Melo 두 어댑터 모두 PRD §4-1 미달 시에만 별도 ADR로 추가 모델(CosyVoice2, Piper-ko 등) 비교. 사전 작성하지 않음. **현 시점 Melo 140ms로 충족.**

### config 예시 (2026-06-02 정정)

```yaml
tts:
  active: "qwen3"                   # "qwen3" | "melo" 토글
  qwen3:
    model_id: "Qwen/Qwen3-TTS-12Hz-1.7B-Base"
    attn_implementation: "sdpa"     # 시도 → 미지원 시 "eager" 폴백
    device_map: "cuda:0"
    dtype: "bfloat16"
    ref_audio_path: "data/ref_voice.wav"
    ref_text: "..."                  # ref WAV 전사 (정확도 향상)
    language: "Korean"               # qwen_tts language 코드
    timeout_sec: 60.0
  melo:
    language: "KR"                  # melo.api TTS language 코드
    device: "cuda:0"
    speaker_id: 0                   # preset 화자
    speed: 1.0
    timeout_sec: 30.0
```

### Pipecat 어댑터 골격 (ADR-012 상속, 2026-06-02 정정)

```python
# app/pipecat_services/qwen3_tts_service.py
from pipecat.services.tts_service import TTSService
from pipecat.frames.frames import TTSAudioRawFrame
from app.adapters.tts.qwen3_client import Qwen3TTSClient
from app.utils.latency import LatencyTracker

class Qwen3TTSService(TTSService):
    def __init__(self, client: Qwen3TTSClient):
        super().__init__(sample_rate=24000)
        self._client = client

    async def run_tts(self, text: str, context_id: str):
        first = True
        tracker = LatencyTracker("tts.first_chunk")
        tracker.__enter__()
        async for chunk in self._client.stream(text):
            if first:
                tracker.__exit__(None, None, None)
                first = False
            yield TTSAudioRawFrame(audio=chunk, sample_rate=24000, num_channels=1, context_id=context_id)
```

`Qwen3TTSClient.stream()`은 LLM-side에서 sentence가 들어오는 순서대로 동기 합성하여 PCM bytes를 yield한다. Pipecat의 SentenceAggregator는 LLM과 TTS service 사이에 배치한다.

## 결과

### 긍정
- v1 50초 → **목표 <500ms 첫 청크** (SDPA + streaming) — PRD v4 §4-1 충족
- Apache 2.0 — XTTS CPML 부담 제거, 외부 배포 시점에 라이선스 재검토 불필요
- 24kHz 고품질 (XTTS와 동급)
- 기존 `data/ref_voice.wav` 그대로 재사용 (6초+ 충족)
- Pipecat sentence-streaming 패턴이 ADR-013 능동 코치 JSON 응답(`text` 필드 sentence 분할)과 자연 결합

### 부정
- transformers + bfloat16 + SDPA 설정 자체는 표준이나, RTX 5090 + Windows 환경에서 사용자 검증 필요
- 모델 다운로드 ~3.4GB (HF 캐시, 최초 1회)
- Qwen3-TTS upstream에 public token-level streaming API 없음 → sentence-batch streaming으로 제약. 첫 청크 지연이 곧 첫 sentence 합성 시간 (PRD 충족 가능성은 짧은 sentence 강제로 확보)
- MeloTTS 병행 어댑터는 보이스 클로닝 미지원 — preset speaker로 제한됨

## 마이그레이션 (v1 base → v3)

1. v1 base의 `app/adapters/tts/qwen3.py` 어댑터 진입부에 `attn_implementation="sdpa"`, `device_map="cuda:0"`, `torch_dtype=torch.bfloat16` 옵션 추가 (클래스명 `Qwen3TTSAdapter` → `Qwen3TTSClient`로 변경, ADR-012)
2. ADR-012 Pipecat `TTSService` 상속한 새 `Qwen3TTSService` 작성 (`app/pipecat_services/qwen3_tts_service.py`)
3. `data/ref_voice.wav` 그대로 재사용 (외부 배포 전 라이선스 클린 ref로 교체)
4. Pipecat `SentenceAggregator`를 LLM과 TTS 사이에 배치하여 sentence-streaming 활성화
5. 사용자 청취 검증: 첫 청크 지연·자연스러움·발음 정확도

## 잔여 사용자 검증

- 한국어 발음 자연스러움 (특히 영문 단어·숫자 발음)
- 첫 청크 지연 < 500ms 측정 (실제 RTX 5090 환경)
- 보이스 클로닝 결과 만족도 (ref WAV 적합성)
