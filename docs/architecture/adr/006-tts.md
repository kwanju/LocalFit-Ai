# ADR-006: TTS — Qwen3-TTS + SDPA 어텐션 + sentence-streaming

- **상태**: Accepted (2026-05-31)
- **관련 ADR**: ADR-002 (로컬-only), ADR-011 (Pipecat), ADR-012 (도메인 어댑터), ADR-013 (능동 코치 지연 의존성)
- **v1 이력 보존**: `_archive/v1/006-tts.md` — Kokoro → Qwen3-TTS → XTTS v2 3차 개정 회고

## 컨텍스트

### v1·v2 시점의 실패 회고

| 시점 | 결정 | 결과 |
|---|---|---|
| v1 초안 | Kokoro 단독 | 한국어 미지원으로 폐기 |
| v1 1차 | Kokoro + Qwen3-TTS 교체형 | Kokoro 한국어 fork 부재로 무의미 |
| v1 2차 | Qwen3-TTS 단독 | RTX 5090에서 합성 자체는 동작 |
| v1 Phase 6 | `config.tts.active = "qwen3"` 출시 | 사용자 청취 시 합성 지연 **~50초** 측정 |
| v2 시점 | XTTS v2 전환 (CPML 라이선스 수용) | 한국어 품질 미달 + Windows 빌드 까다로움 + 라이선스 외부 배포 불가 |

### 진짜 원인 — flash-attn 부재 + SDPA 미설정

v3-rewrite 사전 조사에서 결정적 사실 확인:

- **Qwen3-TTS는 Dual-Track hybrid streaming 아키텍처를 가지며 end-to-end 합성 지연이 97ms까지 가능** (공식 모델 카드).
- 단, **flash-attn 2가 공식 권장이지만 Windows에서 빌드 불가**.
- 커뮤니티 fork는 PyTorch 내장 `SDPA(Scaled Dot Product Attention)`로 대체 (`--no-flash-attn`) — Windows 10/11에서 정상 동작.
- v1 reference 스크립트가 `attn_implementation=None`(no SDPA) + `device_map=None`(CPU 폴백) 기본값을 사용 → 50초 지연 발생.
- 즉 **v1 50초 지연은 Qwen3-TTS 자체 한계가 아니라 잘못된 설정의 결과**다. CPML 라이선스 부담을 안고 XTTS로 갈 이유가 없다.

### Pipecat 통합

Pipecat은 Qwen3-TTS 공식 어댑터를 제공하지 않는다. `pipecat.services.tts.base.TTSService`를 상속해 자체 어댑터를 작성한다(ADR-012). 그러나 어댑터는 얇은 래퍼(transformers.AutoModel + inference) — Pipecat의 frame 처리·인터럽트·sentence aggregation은 부모 클래스가 제공한다.

## 결정

### 기본 TTS = Qwen3-TTS (Apache 2.0)

- **모델**: `Qwen/Qwen3-TTS-12Hz-1.7B-Base` — **Base 채택 사유 = 보이스 클로닝 zero-shot 지원**. non-Base 변형은 프리셋 화자만 지원하므로 채택하지 않음
- **라이선스**: Apache 2.0 — 상업·외부 배포 모두 OK (XTTS CPML 부담 없음)
- **어텐션 구현**: `attn_implementation="sdpa"` — PyTorch 내장 SDPA 사용 (flash-attn 불필요)
- **디바이스 매핑**: `device_map="cuda:0"` (또는 `cuda` — 모든 텐서 GPU 강제)
- **dtype**: `torch.bfloat16` (RTX 5090 + Ampere/Ada/Blackwell 지원)
- **샘플레이트**: 24kHz mono (Qwen3-TTS 출력 그대로)

### Sentence-streaming (지연 단축 핵심)

PRD v4 §4-1 목표(C2C 0.8~1.5s, S2S 2~3s) 충족 전략:

1. **LLM 응답을 sentence 단위로 절단** → 첫 sentence 합성 즉시 재생 시작 (다음 sentence는 백그라운드 합성).
2. **Pipecat의 `SentenceAggregator` + custom `Qwen3TTSService.run_tts(text)` 결합** — frame 단위로 sentence 흘려보냄.
3. 평균 4문장 응답 기준 첫 청크 지연 < 500ms 목표 (LLM stream 시작 + 첫 sentence boundary 도달 시점부터 측정).

### 음성 클로닝 정책

- `ref_audio_path` 1회 캐싱 (`get_conditioning_latents` 결과를 `(speaker_embedding, conditioning_latent)` tuple로 메모리 상주)
- 첫 합성 후 latent 재사용 — 합성 외 오버헤드 0
- ref WAV는 6~10초 한국어 단일 화자 (`data/ref_voice.wav`). 외부 배포 전환 시 라이선스 클린한 ref로 교체.

### 지연 폴백 정책

Qwen3-TTS + SDPA 설정이 사용자 환경에서 지연 목표 미달 시 — **별도 fallback 어댑터를 미리 작성하지 않는다 (YAGNI)**. 미달 사례가 phase-3에서 실제 측정되면 그 시점에 별도 ADR로 fallback 모델 비교·결정. 미사용 어댑터를 미리 작성하면 transformers·Pipecat 버전 업그레이드 시 dead code 유지 비용이 발생한다.

### config 예시

```yaml
tts:
  qwen3:
    model: "Qwen/Qwen3-TTS-12Hz-1.7B-Base"
    attn_implementation: "sdpa"     # ★ 핵심 — flash-attn 불필요, Windows OK
    device_map: "cuda:0"
    dtype: "bfloat16"
    ref_audio_path: "data/ref_voice.wav"
    language: "ko"
    streaming: true
    sentence_split: true            # Pipecat SentenceAggregator 활용
```

### Pipecat 어댑터 골격 (ADR-012 상속)

```python
from pipecat.services.tts.base import TTSService
from pipecat.frames.frames import TTSAudioRawFrame

class Qwen3TTSService(TTSService):
    def __init__(self, config: Qwen3TTSConfig):
        super().__init__(sample_rate=24000)
        self.model = AutoModel.from_pretrained(
            config.model,
            attn_implementation="sdpa",     # 핵심
            device_map="cuda:0",
            torch_dtype=torch.bfloat16,
        )
        self._latents = self._compute_ref_latents(config.ref_audio_path)

    async def run_tts(self, text: str) -> AsyncIterator[TTSAudioRawFrame]:
        # transformers streaming generator → frame 단위 yield
        for chunk in self.model.generate_stream(text=text, latents=self._latents):
            yield TTSAudioRawFrame(audio=chunk.tobytes(), sample_rate=24000, num_channels=1)
```

## 결과

### 긍정
- v1 50초 → **목표 <500ms 첫 청크** (SDPA + streaming) — PRD v4 §4-1 충족
- Apache 2.0 — XTTS CPML 부담 제거, 외부 배포 시점에 라이선스 재검토 불필요
- 24kHz 고품질 (XTTS와 동급)
- 기존 `data/ref_voice.wav` 그대로 재사용 (6초+ 충족)
- Pipecat sentence-streaming 패턴이 ADR-013 능동 코치 JSON 응답(`text` 필드 sentence 분할)과 자연 결합

### 부정
- transformers + bfloat16 + SDPA 설정 자체는 표준이나, RTX 5090 + Windows 환경에서 사용자 검증 필요
- 모델 다운로드 ~3.4GB (HF 캐시, 최초 1회)
- Qwen3-TTS는 transformers `generate_stream` 같은 API가 직접 노출 안 됨 — 자체 generator 작성 필요. 다만 모델 카드에 streaming 예제 있음
- 지연 목표 미달 시 즉시 폴백 어댑터가 없음 — phase-3 실측 후 별도 ADR로 대응 (예상 시나리오는 아니지만 명시)

| 후보 | 탈락 사유 |

| Edge TTS (Microsoft) | 네트워크 의존 — ADR-002 위반 |
| Piper-ko | 한국어 모델 한정, 품질 Qwen3 아래 |
| CosyVoice2 | 0.5B로 가벼움·streaming 가능하지만 한국어 검증 사례 부족, 라이선스 CC-BY-NC |
| ChatTTS | 한국어 미지원 |

## 마이그레이션 (v1 base → v3)

1. v1 base의 `app/adapters/tts/qwen3.py` 어댑터 진입부에 `attn_implementation="sdpa"`, `device_map="cuda:0"`, `torch_dtype=torch.bfloat16` 옵션 추가 (클래스명 `Qwen3TTSAdapter` → `Qwen3TTSClient`로 변경, ADR-012)
2. ADR-012 Pipecat `TTSService` 상속한 새 `Qwen3TTSService` 작성 (`app/pipecat_services/qwen3_tts_service.py`)
3. `data/ref_voice.wav` 그대로 재사용 (외부 배포 전 라이선스 클린 ref로 교체)
4. Pipecat `SentenceAggregator`를 LLM과 TTS 사이에 배치하여 sentence-streaming 활성화
5. 사용자 청취 검증: 첫 청크 지연·자연스러움·발음 정확도

## 잔여 사용자 검증

- 한국어 발음 자연스러움 (특히 영문 단어·숫자 발음)
- 첫 청크 지연 < 500ms 측정 (실제 RTX 5090 환경)
- 보이스 클로닝 결과 만족도 (ref WAV 적합성)
