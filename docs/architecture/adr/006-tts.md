# ADR-006: TTS 교체형 어댑터 — Kokoro + Qwen3-TTS

- **상태**: Accepted (Revised 2026-05-21)
- **이력**: 초기 Kokoro 단독 → 교체형 다중 구현체로 개정
- **관련 ADR**: ADR-004 (LLM 듀얼, 동일 철학), ADR-010 (어댑터 인터페이스)

## 컨텍스트

PRD 미결 사항 #1: "Kokoro vs Coqui — 톤 파라미터 비교 후 결정". 2026-05 재검토 결과 Qwen3-TTS(2026-01 출시, Apache 2.0)가 한국어를 명시적으로 지원하며 ADR-004의 Qwen 3.5 LLM과 같은 생태계라는 이점이 있다.

한국어 TTS 품질은 직접 청취 전엔 확정 불가하므로, 여러 구현체를 두고 설정으로 전환하는 방식을 채택한다. **동시 사용이 아니라 교체형이다** (활성 1개만 로드).

## 결정

### 교체형 어댑터 구조

```
app/adapters/tts/
├── protocol.py          # TTSAdapter Protocol (ADR-010)
├── kokoro.py            # KokoroAdapter
├── qwen3.py             # Qwen3TTSAdapter
└── __init__.py          # config 기반 어댑터 선택 (dict 매핑)
```

팩토리 클래스 대신 dict 매핑으로 단순화:

```python
# app/adapters/tts/__init__.py
_REGISTRY = {"kokoro": KokoroAdapter, "qwen3": Qwen3TTSAdapter}

def get_tts_adapter(config) -> TTSAdapter:
    return _REGISTRY[config.tts.active]()
```

### 설정

```yaml
# config.yaml
tts:
  active: "qwen3"           # "kokoro" | "qwen3"
  fallback: "kokoro"
  voice: "default"
  speed: 1.0
```

### 1차 구현체
- **Kokoro-82M**: 작고 빠름 (~300MB, ~0.5GB VRAM)
- **Qwen3-TTS 0.6B**: 한국어 명시 지원, 97ms 지연 (~1.5GB VRAM)

### 청취 비교 (P0 첫 주)
두 구현체를 모두 만든 뒤 한국어 30문장으로 청취 비교 → config.yaml의 active 값으로 1차 모델 확정. 마음에 안 들면 언제든 설정만 변경 후 재시작.

### 동시 사용 안 함
활성 모델만 메모리 로드. 동시 다중 TTS(페르소나별 음색)는 PRD 부록 C와 함께 P2에서 검토하되, 현재 교체형 구조가 그 확장을 막지 않는다.

## 결과

### 긍정
- ADR-004 LLM 듀얼과 동일 철학 → 코드 일관성
- 청취 비교로 데이터 기반 결정
- 설정 한 줄로 전환, 동시 메모리 점유 없음
- Qwen3-TTS는 Apache 2.0 (안전한 라이선스)

### 부정
- 구현체 2개 = 유지보수 표면적 증가 (라이브러리 버전 호환 등)
- Qwen3-TTS는 출시 5개월 — 운영 중 버그 발견 가능성
- 구현체별 모델 다운로드 필요 (ADR-013 반영)

## 대안

| 후보 | 탈락 사유 |
|---|---|
| Kokoro 단독 (초기 결정) | 한국어 명시 지원 불확실, 비교 없이 확정 부담 |
| Coqui XTTS-v2 | 회사 폐업(2024), 신규 업데이트 없음 — 백업으로만 기록 |
| 동시 다중 TTS (페르소나별) | PRD 부록 C가 P2, MVP는 페르소나 1종이라 불필요 |
| 구현체 3개 이상 | 유지보수 표면적 과대, 2개로 충분 |
