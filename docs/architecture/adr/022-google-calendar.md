# ADR-022: Google Calendar 연동 (OAuth2 Desktop, 외부 API)

- **상태**: Proposed (2026-06-10)
- **관련 ADR**: ADR-002 (단일 사용자, 로컬-only 완화), ADR-027 (알림/스케줄러), ADR-021 (v4 스코프)

## 컨텍스트

v4 는 백그라운드 스케줄러로 운동 시간을 리마인드한다(ADR-027). 운동 일정을 어디에 저장·관리할지가 갈린다. deep interview(§6-2)에서 사용자는 **앱 내장 로컬 스케줄만**이 아니라 **Google Calendar 연동**을 선택했다 — 폰 등 다른 기기와 동기화되고, 미팅 같은 다른 일정을 읽어 틈새 시간을 추천할 수 있는 가치 때문.

이는 v3 의 로컬-only 원칙(ADR-002)과 긴장한다(외부 API + OAuth). 단일 사용자 본인 계정 1개에 한해 완화한다.

## 결정

- **Google Calendar API 연동**, **OAuth2 Desktop client** 흐름.
  - 최초 1회 브라우저 동의 → refresh token 을 **로컬에 안전 저장**(OS 자격증명 저장소 또는 암호화 파일). 이후 자동 갱신.
- **운동 일정 등록**: 코치가 일정을 캘린더에 등록할 때 **사용자 확인 후** 기록(자동 등록 시 동의 단계).
- **다른 일정 읽기**: 미팅 등 기존 일정을 읽어 "오늘 저녁 8시까지 비어 있으니 그때 어때요?" 같은 틈새 추천.
- **로컬-only 완화 범위**: 캘린더 연동에 한함. 본인 계정 1개. 다른 데이터(운동 기록·메모리·음성)는 로컬 유지(ADR-002).
- **안 행복한 경로** (회고 원칙 1):
  - 캘린더 없는/연동 거부 사용자 → **로컬 스케줄 fallback**(ADR-027)로 동작, 캘린더 기능만 비활성.
  - 토큰 만료/네트워크 오프라인 → 캘린더 기능 graceful degrade, 로컬 스케줄·코칭은 계속.

## 결과

### 긍정
- 폰 등 기기 간 일정 동기화
- 기존 일정 기반 틈새 추천 가능
- 단일 사용자라 OAuth 흐름 단순(계정 1개)

### 부정
- 외부 API 의존 — 오프라인/장애 시 캘린더 기능 불가(fallback 필요)
- OAuth 토큰 보관 = 로컬-only 원칙의 부분 완화(보안 저장 책임)
- Google API 정책/쿼터 변경 리스크

## 대안

| 후보 | 탈락 사유 |
|---|---|
| 앱 내장 로컬 스케줄만 | 폰 동기화·외부 일정 읽기 불가 — 사용자가 Calendar 선택 |
| CalDAV 범용 연동 | 단일 사용자엔 과한 일반화. Google 직접 연동이 단순 |
| 자동 일정 등록(확인 없음) | 회고 "LLM 마음대로" 경고 — 등록은 확인 단계 필수 |
