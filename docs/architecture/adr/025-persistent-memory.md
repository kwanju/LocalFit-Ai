# ADR-025: 영속 메모리 — SQLite 구조화 + 자유텍스트, 부상/제약 전량 주입

- **상태**: Proposed (2026-06-10)
- **관련 ADR**: ADR-008 (SQLite + SQLModel), ADR-013 (능동 코치), ADR-023 (컨디션), ADR-024 (플랜), ADR-028 (첫 체력검증)

## 컨텍스트

v3 는 WebSocket 세션 안에서만 맥락을 기억하고, 세션이 끝나면 LLM history/slot 을 비운다. 그래서 사용자가 한 번 말한 선호·제약을 영구히 기억하지 못한다.

v3 회고(2026-06-07)의 구체 사례: 사용자의 **"왼쪽 어깨가 안 좋음"** 같은 정보를 기억하지 못해 안전에 직결되는 제약을 놓쳤다. 부상/제약은 **누락 시 사고로 이어지는 안전 정보**다.

전제:
- 단일 사용자(ADR-002) — 데이터량이 작다. 사용자 발화/선호/제약 전부를 매 세션 프롬프트에 **전량 주입할 수 있는 규모**.
- v4 비전: 첫 체력검증(ADR-028) 결과를 기준선으로 저장하고, 컨디션(ADR-023)·플랜(ADR-024)이 이를 참조한다.

## 결정

**2층 메모리 구조.** 안전·기준선은 결정적으로, 부드러운 선호는 느슨하게 분리한다.

### 1층 — 구조화 필드 (안전·기준선, 전량 주입)

SQLModel 테이블로 저장하고 **매 세션 시작 시 전량 프롬프트 주입**(검색에 의존하지 않음):

- `injuries` / `constraints` — 부상·제약 (예: "왼쪽 어깨", "무릎 통증"). **안전 가드의 입력**.
- `fitness_baseline` — 첫 체력검증(ADR-028) 결과 (종목별 최대치·시작 강도). 플랜(ADR-024)의 기준선.
- 핵심 `preferences` 중 구조화 가능한 것 (예: 선호 운동 시간대)

### 2층 — 자유텍스트 메모 (부드러운 선호)

- LLM 이 대화 중 추출한 자유로운 선호/맥락을 텍스트로 누적 (예: "아침 운동을 선호함", "스쿼트를 싫어함").
- `memory_facts` 테이블 (text + created_at + tags).
- 검색 = **최근순 / 키워드** (벡터 검색 아님). 주입 시 토큰 예산 내 최근·관련 N건.

### 미도입 — vector DB

- chromadb 등 vector DB 는 도입하지 않는다. 1인 데이터량에 과잉(YAGNI)이고, 의미 검색은 **비결정적이라 안전 정보(부상)가 검색에서 누락될 위험**이 있다. 안전 정보는 1층(전량 주입)이 책임진다.

### 주입 책임 분리

- 부상/제약(1층)은 **절대 검색 결과에 의존하지 않고 항상 전량** 프롬프트에 들어간다.
- 자유텍스트(2층)만 검색·요약 대상. 2층이 비거나 검색이 빗나가도 안전에는 영향 없음.

## 결과

### 긍정
- **안전 정보 누락 0** — 부상/제약은 매번 전량 주입 (회고 사례 직접 해결)
- 단순·예측 가능·디버깅 쉬움 (구조화 필드 = SQL 조회로 확인)
- 1인 규모라 전량 주입이 토큰상 현실적
- ADR-028(체력검증)·023(컨디션)·024(플랜)의 공용 기준선 저장소

### 부정
- 자유텍스트 메모가 늘면 2층 검색에서 일부 누락 가능 (단 안전 정보는 1층이라 무관)
- 전량 주입으로 프롬프트 길이 증가 — 토큰 예산 관리 필요
- "무엇을 1층 구조화로 승격할지" 판단 로직 필요 (LLM 추출 + 안전 키워드 규칙)

## 대안

| 후보 | 탈락 사유 |
|---|---|
| 구조화 필드만 | 스키마에 없는 뉘앙스("아침엔 무릎이 뻑뻑함") 손실 |
| vector DB(RAG) | 1인 데이터에 과잉 + 임베딩 의존성 + **안전 정보 검색 누락 위험** |
| 세션 history 만 (v3 현행) | 영속성 없음 — 회고가 지적한 바로 그 문제 |
