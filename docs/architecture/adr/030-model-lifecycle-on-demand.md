# ADR-030: 모델 lifecycle — on-demand 로드/언로드

- **상태**: Proposed (★ 탐사 의존 — Phase v4-0 S-8 통과 전제)
- **Supersedes**: ADR-015 (모델 동시 상주 — keep_alive 24h)
- **관련 ADR**: ADR-003 (Ollama), ADR-005 (STT), ADR-006 (TTS), ADR-029 (LLM 모델)

## 컨텍스트

ADR-015 는 세 모델(LLM+STT+TTS, ~10GB)을 GPU 에 **24시간 상주**시킨다. ADR-015 자신이 부정 항목에 *"미사용 시간에도 GPU 메모리 ~10GB 점유 — 다른 작업 동시 사용 시 불편"*을 인정했고, 대안표에서 "lazy load(요청 시 로드)"를 *"첫 요청 30~60초 지연 → UX 불가"*로 탈락시켰다.

v4 사용자 요구(§6-8)가 이 트레이드오프를 뒤집는다:
- 사용자는 **게임을 켠 채** 같은 머신에서 운영하고 싶어 한다.
- 16GB VRAM 베이스라인(친구 공유, §6-1)에서는 모델 상주 시 게임과 VRAM 경합.
- → **"게임 VRAM 공존"이 "콜드스타트 0"보다 중요**하다. 콜드스타트는 "코치 준비 중" UX 로 흡수 가능.

## 결정

- **무거운 모델(STT+TTS+LLM)은 상주하지 않는다.** 운동/대화 세션 시작 시 GPU 로드, 세션 종료 시 언로드.
  - **LLM**: Ollama `keep_alive=0`(또는 짧게) — 요청 후 자동 언로드.
  - **STT/TTS**: Python 모델 객체 해제 + `torch.cuda.empty_cache()`로 VRAM 반환.
- **백그라운드 상주 = 경량만**: Tauri 트레이 + 스케줄러 + 경량 Python(모델 없음). 평소 VRAM ≈ 0.
- **콜드스타트 흡수**: 세션 시작 → 첫 응답 전 "코치 준비 중" UX. (실측치는 Phase v4-0 S-8 에서 기록)
- **16GB 베이스라인**: 세션 중 ~13GB(STT+TTS+qwen3.5:9b). 세션 밖에서는 게임 등이 16GB 자유 사용. 32GB(RTX 5090)는 세션+게임 동시 여유.
- **ADR-015 무효화**: `keep_alive: 24h`, FastAPI lifespan warmup 상주 → 폐기.
- **안 행복한 경로**:
  - VRAM 부족(타 앱이 점유) → 로드 실패 시 "게임/다른 앱을 닫아주세요" 안내.
  - **prewarm 미채택**: 스케줄러가 미리 로드하면 게임과 VRAM 충돌 → 요청(세션 시작) 시 로드.

## 결과

### 긍정
- 평소 VRAM 해방 — 게임 등 타 앱과 공존(사용자 핵심 요구)
- 16GB~32GB 하드웨어 스펙트럼 흡수
- 백그라운드 점유 최소

### 부정
- **콜드스타트 지연** — 세션 시작 시 모델 로드 대기(UX 로 흡수)
- 로드/언로드 코드 + VRAM 반환 검증 필요(메모리 누수 주의)
- 잦은 세션 전환 시 반복 로드 비용

## 탐사 의존 / no-go 대응

- **의존**: Phase v4-0 S-8 — 세션 종료 후 `nvidia-smi`로 VRAM 반환 확인 + 콜드스타트 실측.
- **no-go 시**(언로드가 깨끗이 안 되거나 콜드스타트가 비현실적): ADR-015(상주) 유지 + 32GB 권장 사양으로 후퇴, 16GB 베이스라인 재검토.

## 대안

| 후보 | 탈락 사유 |
|---|---|
| keep_alive 24h 상주 (ADR-015) | 게임 등 타 앱과 VRAM 공존 불가 — v4 사용자 요구 미달 |
| 세션 전 prewarm | 게임이 VRAM 점유 중이면 충돌 — 요청 시 로드가 안전 |
| LLM만 상주, STT/TTS lazy | 부분 점유로 16GB 게임 공존 여전히 빡빡 |
