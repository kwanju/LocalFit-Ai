# ADR-027: 능동 알림 + 백그라운드 스케줄러

- **상태**: Proposed (★ 탐사 의존 — Phase v4-0 S-6/S-7 통과 전제)
- **관련 ADR**: ADR-031 (native Tauri — 알림 채널), ADR-022 (캘린더 — 일정 소스), ADR-021 (세션 모델 + 스케줄러)

## 컨텍스트

v3 는 사용자가 앱을 켤 때만 동작한다. v4 는 운동 시간/체크인을 **능동적으로 리마인드**한다(§6-3). 단 ADR-021 대로 "선제 발화 상시 에이전트"가 아니라 **스케줄 기반 리마인드**다.

PWA(브라우저 Notification)는 브라우저가 닫히면 알림이 오지 않는다. 그래서 **native toast + 트레이 상주**가 필요하고, 이는 ADR-031(Tauri)에 의존한다.

## 결정

- **백그라운드 스케줄러**: Tauri 트레이에 상주하는 경량 프로세스가 운동 일정(ADR-022 캘린더 또는 로컬 스케줄)을 읽어 알림을 발생.
- **Native toast**(Windows): 운동 시간 도래 / 체크인 리마인드. **클릭 시 앱 포커스 + 세션 시작 흐름**으로 진입.
- **알림 정책**:
  - 시점: 운동 시간 기준(사용자 설정, 예: N분 전).
  - 음소거 시간대(예: 야간) 설정으로 알림 억제.
- **선제 발화 아님**: 코치가 먼저 말 거는 게 아니라 스케줄 기반 리마인드만(ADR-021).
- **모델 비상주**: 스케줄러/트레이는 경량(모델 없음). 알림 클릭 후 세션 시작 시점에 모델 로드(ADR-030).
- **안 행복한 경로**:
  - OS 알림 꺼짐(집중 모드 등) → 앱 내 배지/리스트로 fallback.
  - 스케줄러 프로세스 사망 → 앱 복귀 시 누락 리마인드 표시.
  - 캘린더 미연동 → 로컬 스케줄 기반 알림(ADR-022 fallback).

## 결과

### 긍정
- 앱을 안 켜도 운동 시간 리마인드 — 지속 코치의 핵심 UX
- 알림 → 즉시 세션 시작으로 마찰 최소

### 부정
- native(Tauri) 의존 — 탐사 no-go 시 기능 축소
- OS 알림 설정/권한에 종속
- 백그라운드 상주 프로세스 관리(좀비/중복 방지)

## 탐사 의존 / no-go 대응

- **의존**: Phase v4-0 S-6(toast)·S-7(트레이 상주) 통과.
- **no-go 시**: PWA Notification API(브라우저 열려 있을 때만)로 축소하고, 능동 알림 범위를 "앱 사용 중 리마인드"로 한정. 비전 §0-2 수정.

## 대안

| 후보 | 탈락 사유 |
|---|---|
| PWA Notification 만 | 브라우저 닫으면 알림 없음 — 능동 알림 목적 미달 |
| 선제 발화 상시 에이전트 | ADR-021 스코프 밖(세션 모델 유지) |
| OS 작업 스케줄러(Windows Task Scheduler) 직접 | 앱 상태/세션 연계 어려움 — 트레이 상주가 통합적 |
