# ADR-004: 듀얼 LLM 모델 — Qwen 3.5 9B + Gemma 4 E4B

- **상태**: Accepted (Revised 2026-05-27)
- **관련 ADR**: ADR-003 (Ollama 런타임), ADR-010 (어댑터 인터페이스), ADR-013 (모델 다운로드)
- **이력**: 초기 결정은 Qwen 2.5 7B 단독이었으나, Gemma 4 출시(2026-04)와 사용자 요청에 따라 듀얼 구조로 개정. 2026-05-27 정정: `qwen3.5:7b`는 Qwen 3.5 라인업에 존재하지 않는 태그(3.5는 0.8b/2b/4b/9b/27b/35b/122b)이므로 대화용을 `9b`로 정정하고 경량 fallback `4b`를 명시

## 컨텍스트

한국어 코칭 품질과 추론 깊이는 모델마다 trade-off가 있으며, PoC 없이 한쪽으로 확정하기엔 위험이 있다. RTX 5090(32GB VRAM)과 디스크 여유로 두 모델을 동시 보유해도 비용이 없다. ADR-010의 LLM 어댑터 인터페이스가 이미 모델 교체를 추상화하고 있다.

## 결정

### 1차 모델 라인업
- **Qwen 3.5 9B** — 한국어 우선 코칭 대화(기본). Ollama: `qwen3.5:9b` (= `qwen3.5:latest`, 6.6GB). 경량 fallback으로 `qwen3.5:4b`(3.4GB)도 사용 가능
- **Gemma 4 E4B** — 추론 깊이 필요한 경우(일정 분석 등). Ollama: `gemma4:e4b`

E4B를 선택한 이유는 4B 파라미터로 Qwen 9B와 함께 5090(32GB)에 부담 없이 상주 가능하며, Gemma 4의 함수 호출·추론 강점을 확인하기에 충분하기 때문이다. 26B A4B는 P1에서 검토한다.

### 전환 방식
**설정 파일 기반 단일 활성 모델**.

```yaml
# config.yaml
llm:
  active_model: "qwen3.5:9b"      # 대화 기본 / 또는 "gemma4:e4b"
  fallback_model: "qwen3.5:4b"    # 경량 fallback (또는 "gemma4:e4b")
  keep_alive: "1h"
```

런타임 전환은 백엔드 재시작으로 처리하며, P1에서 UI 토글 추가를 고려한다.

### keep_alive 관리
활성 모델만 메모리 상주, 비활성 모델은 디스크에만 존재. Ollama가 첫 호출 시 자동 로드, idle 시 자동 unload.

### Qwen 3.5 thinking 모드
Qwen 3.5는 기본적으로 thinking 모드가 활성화되어 `message.content`가 빈 문자열로 반환될 수 있다.
`LLMRequest.think: bool = False`(기본값)로 비활성화하며, 복잡한 추론이 필요한 경우에만 `think=True`를 명시한다.

### 회고 시점
**MVP 사용 10회 운동 세션 후** 듀얼 운영 데이터를 보고 단일 모델로 좁힐지 또는 의도별 라우팅 규칙을 정할지 결정한다. 결정 미루기 함정을 막기 위해 명시적 트리거를 설정한다.

## 결과

### 긍정
- PoC 부담 없이 두 모델 즉시 사용 가능
- 설정 한 줄 변경 + 재시작으로 전환 — 어댑터 코드 무변경
- 동시 메모리 점유 없음 — 5090에서도 자원 효율적

### 부정
- 디스크 ~8GB 추가 소비 (5090 환경에서 부담 없음)
- 모델 전환 시 첫 호출 5~10초 지연
- "둘 다 받아두고 결정 안 함" 함정 우려 → 10회 회고 트리거로 방지

## 대안

| 후보 | 탈락 사유 |
|---|---|
| 두 모델 동시 메모리 상주 + 의도별 라우팅 | PRD 시나리오에 불필요, 오버엔지니어링 |
| 한 모델만 채택 | PoC 부담 회피하려는 듀얼 결정 자체와 충돌 |
| 모델마다 별도 어댑터 클래스 | Ollama가 둘 다 같은 API로 노출 — 단일 어댑터 + 모델명 파라미터로 충분 (YAGNI) |
| Qwen 2.5 7B (초기 결정) | 한 세대 뒤떨어짐, Qwen 3.5로 업그레이드 |
