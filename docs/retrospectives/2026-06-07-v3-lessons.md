# v3 회고 — 다음 프로젝트에 적용할 원칙 (2026-06-07)

> ADR 19개 + PRD v4 + phase 8단계까지 적었는데, 결국 사용자가 직접 클릭하며 잡아낸 버그가
> 줄줄이 나왔다. 무엇이 문제였나, 어떻게 하면 줄일 수 있나.

## 1. 사실 관계 — 무엇이 실제로 일어났나

각 phase의 자동 테스트는 다 통과했다. 그런데 사용자가 직접 써보면 다음 버그가 줄줄이 발견됐다:

- `eunjeon` 미설치 → MeloTTS 첫 합성 실패 (g2pkk 의존성)
- Pipecat 1.3.0 TTSSettings/STTSettings `NOT_GIVEN` 에러
- Pipecat 1.3.0 `BaseOutputTransport.write_transport_frame()` no-op → `TextFrame` 이 WebSocket 으로 안 나감
- `push_start_frame=True` + `stop_frame_timeout_s` 조정 안 하면 audio context drop
- 서비스 싱글톤 (TTS/STT) 재링크 race → 모드 전환마다 파이프라인 막힘
- UI `<audio>` 엘리먼트 chunk 마다 재시작 → 코치 음성 끊김
- 카운터(즉시) vs 음성(TTS 5초+) sync 불일치
- LLM 이 사용자 확답 없이 `start_counting` 발행 → 운동 강제 시작
- WS disconnect 후 LLM 응답이 도착해서 운동 시작 처리
- BERT 모델 lazy 로드 + 동시 호출 race → meta tensor 에러
- 신규 사용자에게 "지난주처럼" 응답 (테스트 세션이 SetLog 없이 history 로 카운트됨)
- Vite proxy 에 `/api` 누락 → 캘린더 데이터 못 가져옴

## 2. 왜 명세가 있었는데도 막지 못했나

### 2-1. 명세에 "안 행복한 경로" 가 빠져 있었다
ADR/PRD 는 "뭘 만들지" 는 적었지만 "사용자가 실제로 어떻게 쓸지" 는 부족했다. 빠진 것들:
- **사용자 시나리오 디테일** — 신규 사용자 (SetLog 0개), 사용자가 confirm 안 하는 경우, 모드 빠른 전환, 세션 도중 disconnect, 인터럽트 + 재시작
- **세트 단위 운동 계획** — `propose_set.sets` / `propose_set.rest_sec` 필드는 있는데 다회 세트 자동 진행 흐름은 명세에 없음
- **오디오 스트리밍 디테일** — UI가 chunk 를 큐로 잇는 부분 명세 없음
- **데이터 관리** — 초기화/리셋 같은 운영 기능 명세 없음
- **에러 처리/복구** — TTS 실패 시? STT 끊김? LLM 타임아웃?

### 2-2. 의존성 메이저 변경이 명세를 흔들었다
명세 작성 시점 가정 vs 실제:
| 영역 | 명세 가정 | 실제 (Pipecat 1.3.0) |
|---|---|---|
| TextFrame 송신 | transport 가 자동 송신 | `write_transport_frame()` no-op — 별도 처리 필요 |
| TTSService 인자 | `model=`, `voice=`, `language=` 직접 | `settings=TTSSettings(...)` dataclass |
| audio context | 자동 | `push_start_frame=True` 또는 명시 호출 |
| SystemFrame 처리 | 일반 큐 | `process_queue` 우회 → race |
| FrameProcessor 재사용 | OK | 단일 파이프라인 lifecycle 종속 |

→ 메이저 업그레이드는 명세 자체를 흔들 수 있다. 사용자와 의논하여 결정된 부분을 명세에 반영한다.

### 2-3. 테스트가 단위 편중 — E2E 부재
- Unit 254개 다 통과, 단위 로직 단단함
- **"브라우저에서 사람이 클릭하고 음성 듣는" E2E 시나리오는 자동화 없음**
- 결과: 단위 통과해도 실제 사용자 경험은 깨진 채로 통과

### 2-4. 통합 검증 시점이 늦었다
- phase 1~8 다 끝나고 통합 테스트 → 그 시점에 Pipecat 호환성, sync, UI audio 문제 한 번에 폭발
- "각 phase 끝에 사람이 직접 체험" 단계가 없었다 (자가 체크리스트는 있었지만 사용자 체험 단계가 빠짐)

## 3. 다음 프로젝트에 적용할 4가지 원칙

### 원칙 1 — 명세에 "안 행복한 경로" 섹션 강제
모든 ADR/feature spec 에 다음 섹션을 의무 항목으로:

- **에러 시나리오** — 외부 의존성 죽으면? 사용자가 abort 하면? 타임아웃은?
- **신규/빈 데이터 시나리오** — 0개 사용자, 0개 기록일 때 화면이 어떻게 보이나
- **상태 전환** — 모드 전환, 인터럽트, 종료 중 새 이벤트 도착
- **운영 기능** — 리셋, 디버그 페이지, 데이터 export, 로그 보기

### 원칙 2 — phase 끝마다 "사람이 30초 클릭" E2E 시나리오 명시
phase 명세 끝에 `## 검증` 섹션 추가:
```
1. 백엔드 + UI 띄움
2. C2C 모드 클릭 → "안녕" 입력 → 코치 응답 보임 확인
3. 모드 전환 → 다음 세션 시작 확인
4. ...
```
체크 못 한 phase 는 `done` 처리 X. v3 의 phase 자가 체크리스트(A~H) 는 코드 관점이라 사용자 체험 단계가 빠져 있었음.

### 원칙 3 — 의존성 메이저 업그레이드는 별도 "탐사 phase"
새 라이브러리 또는 메이저 업데이트:
1. 작은 prototype 으로 1-2일 탐사 → API 호환성/이슈 발견
2. 발견된 issue 로 ADR 보강 또는 폐기
3. 그 다음 본격 phase 진입

v3 는 Pipecat 1.0대 가정의 명세를 1.3.0 으로 그냥 갔다 — 누적 비용이 컸다.

### 원칙 4 — 테스트 비율 의식 (단위 vs 통합 vs E2E)
- 단위: "내가 짠 로직 검증"
- 통합: "여러 모듈이 어울리는지"
- E2E: "사용자 경험 끝-끝"

세 종류 다 필요. v3 는 단위 편중. 다음엔 phase 끝마다 E2E 시나리오 자동/반자동 실행.

## 4. 무한 오류의 굴레를 피하는 5가지 패턴

**증상**: 버그 하나 잡으면 다른 게 또 튀어나옴 → 끝이 안 보임.

v3 의 흐름이 정확히 이랬다:
> 음 끊김 잡으니 sync 보임 → sync 잡으니 LLM 폭주 보임 → LLM 잡으니 세션 종료 보임 → ...

### 4-1. 표면 vs 근본 — 5분 멈춰서 묻기
버그 잡기 전에 "이게 왜 일어났나" 5분 질문.

- **표면 원인**: 이 줄에 typo, 변수 이름 틀림
- **근본 원인**: 설계 가정이 깨졌음, 라이브러리 호환성, 잘못된 멘탈 모델

근본을 안 잡으면 같은 뿌리에서 또 다른 표면 버그가 나온다.

예: "음 끊김" 표면은 UI 청크 처리, **근본은 Pipecat 1.3.0 audio 출력 모델 미스매치**. 근본을 안 잡았으면 sync 문제가 다시 같은 자리에서 나옴.

### 4-2. 버그 한 박자 모아서 보기
하나 잡고 즉시 commit + push 보다 3-4개 모아서 패턴 보기.

- 3-4개 모이면 "이거 다 같은 뿌리에서 나오네" 발견 자주.
- 발견하면 한 번에 근본 수정 → 다음 버그 봉우리 사전 차단.

### 4-3. 재현 가능 시나리오 → 자동 테스트
매번 사람이 손으로 재현하면 같은 버그 또 들어온다.

- 한 번 잡은 버그는 재현 스크립트 만들고 회귀 방어
- 사용자 보고 버그 → 받자마자 단위/통합 테스트로 박제
- 다음 phase 에서 깨지면 즉시 알람

### 4-4. "그만하기" 결정 의식
모든 엣지 케이스 다 잡으면 끝이 없다.

- 사용자가 실제로 부딪힐 시나리오 한정
- 나머지는 `known issue` 로 문서화 + 후순위
- 이 문서가 우선순위 결정의 근거

### 4-5. diff 작은 변경 + 즉시 검증
큰 PR 로 묶어서 한 번에 합치면 어느 변경이 어느 버그 만든 건지 추적 어렵다.

- 변경 한 종류씩 끊고 사람 검증 한 번씩
- 큰 refactor 는 별도 브랜치 + 별도 검증
- 변경 → 테스트 → 사용자 확인 → 다음 변경
