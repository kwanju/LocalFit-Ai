## 0-2. Deep Interview 진행 현황 (2026-06-10 시작, 계속 갱신)

> 새 세션이 §6 질문 묶음을 한 묶음씩 채우는 중. 확정된 결정만 여기 누적.

**확정 (2026-06-10):**
- **비전 범위** = "운동 + 컨디션/건강". 영양 추적·일과 비서는 **제외**. (§6-1)
- **사용자** = 단일 사용자 유지 (ADR-002 계승). (§6-1)
- **마이그레이션** = **v4 별도 브랜치 `v4-rewrite`** + v3 master 동결 유지. (당초 "master 직접 진화"였으나 Tauri 탐사 등 큰 변경 고려해 번복) (§4)
- **지속성** = v3 세션 모델 유지 + **백그라운드 스케줄러** 추가 (상시 가동 에이전트 아님, 스케줄러가 운동시간/체크인 리마인드만). (§6-3)
- **알림** = **Native (Tauri 등)** — OS toast + 백그라운드 상주. ADR-010(데스크탑 패키징 폐기)을 v4 새 ADR로 **의식적으로 뒤집음**. (§6-3)
- **컨디션** = **자가보고 체크인** (외부 건강 API 없음, 로컬 only). (§6-4)
- **상주 형태** = 미정 (Tauri 트레이만 상주 + Python 사이드카 vs 둘 다 상주) → 탐사 phase 후 결정. (§6-8)
- **탐사 phase 강제** = Tauri 본격 도입 전 1~2일 프로토타입(toast/Python 사이드카/React 웹뷰/빌드)으로 검증. 회고 원칙 3 + v2 패키징 폐기 이력 반영. 되면 ADR 확정, 안 되면 PWA 후퇴. (§6-8)

- **영속 메모리** = **SQLite + 자유텍스트 메모** 하이브리드. 단 **부상·제약은 안전 직결이므로 구조화 필드 + 매 세션 전량 프롬프트 주입**(검색에 맡기지 않음). 부드러운 선호("아침 운동 선호")는 자유텍스트. vector DB는 1인 데이터량에 과잉(YAGNI)으로 배제. (§6-6)
- **운동 플랜** = **주간 목표 + 조정은 확인**. 목표/기준선은 세우되 컨디션에 따른 변경은 코치가 제안 → 사용자 동의(회고 ConfirmRule 정신). 자동 조정은 "LLM 마음대로" 경고로 배제. 구현 순서: v4 초기엔 주간목표 세팅+일자 분배부터, 자동 제안은 컨디션 레이어 안정 후. (§6-5)
- **첫 세션 체력검증(ADR-028)** = **대화형 + 자가보고**. 기존 온보딩 폼 값을 시드로 코치가 대화로 확인/보정 → 보수적 시작(X%). 실측 세트는 부담·부상 위험으로 배제. 재평가 트리거 = N주 경과 또는 "너무 쉬움/어려움" 피드백. (§6-11)
- **데이터 흐름**: 첫 체력검증 → 메모리 구조화 필드(기준선) 저장 → 플랜(주간목표)+기준선 → 컨디션 자가보고와 결합해 "오늘 강도 조절?" 제안(확인). 부상/제약은 전량 주입되어 안전 가드.

- **캘린더** = **Google Calendar 연동** (ADR-022 유지). OAuth2 Desktop client + 외부 API. 운동 일정 등록 + 다른 일정(미팅 등) 읽어 틈새 추천. **로컬-only(ADR-002)를 캘린더에 한해 완화** — 단일 사용자 본인 계정 1개. (§6-2, §6-9 일부 답)
- **운동 종목** = **v3 4종(풀업/푸시업/스쿼트/플랭크) 고정 유지**. 종목 확장·사용자 정의는 후순위. v4는 컨디션·메모리·플랜에 집중. (§6-7)
- **UX 모드** = v3 4모드에서 **S2C(음성입력→텍스트출력) 제거 → 3모드(C2C/C2S/S2S)**. 음성으로 말하고 텍스트로만 답받는 맥락이 어정쩡. (§6-10)

- **음성 파이프라인** = **Pipecat 유지 + 버전 핀**. 자체 구현은 v1에서 폐기한 길(인터럽트·turn·VAD 재발명). v4 신규 작업과 독립적이라 건드릴 이유 없음. (§6-8)
- **에이전트 framework** = **자체 유지(v3 instructor) + tool-use 점진 강화**. LangGraph는 Pipecat 통합 churn 위험+과잉, LlamaIndex는 vector DB 안 써서 부적합. (§6-8)
- **STT/TTS** = 변경 없이 유지 (faster-whisper large-v3-turbo + faster-qwen3-tts 단독). v3에서 막 안정화한 자산. (§6-8)
- **음성 데이터** = **raw 오디오 휘발, 전사 텍스트만 영속**. 메모리는 텍스트면 충분, raw 저장은 용량·프라이버시 부담만(YAGNI). (§6-9)
- **LLM(두뇌) 모델** = **qwen3.5:9b dense** 기본. qwen3:8b 대비 최신·한국어/tool-use 개선, VRAM 차이 미미. **config.yaml로 교체 가능**(ADR-004) — 고사양 사용자는 `qwen3.6:35b-a3b`(MoE, ~21GB) 등으로 상향. ※ LLM(qwen3.5:9b)과 TTS(faster-qwen3-tts)는 이름만 같고 별개 모델·독립 교체. (§6-8)
- **모델 lifecycle = on-demand 로드/언로드** (★ADR-030 발행, **supersedes ADR-015** — v3의 keep_alive 24h 상주를 뒤집음. ADR-015 부정 섹션이 이미 "미사용 시 10GB 점유 불편"을 인정했고, 대안표의 "lazy load"를 콜드스타트로 탈락시켰는데 v4는 게임 VRAM 공존을 우선해 트레이드오프를 역전). 무거운 모델(STT+TTS+LLM)은 **상주 안 함** — 운동/대화 세션 시작 시 GPU 로드, 종료 시 언로드(Ollama `keep_alive=0` + Python `empty_cache()`). 평소엔 게임 등이 VRAM 자유 사용. 백그라운드 상주는 Tauri 트레이+스케줄러+경량 Python만. **trade-off = 콜드스타트 지연** → "코치 준비 중" UX로 흡수. v3의 warmup 상주 모델에서 전환. (§6-3·§6-8 "상주 형태" 해소)
- **하드웨어 베이스라인 = VRAM 16GB**. 모델 예산(~13GB: STT+TTS+9B)을 16GB에 맞춤 → 고사양(32GB, RTX 5090)은 모델+게임 동시 구동 여유. MoE 35B-A3B(~21GB)는 16GB 동시 상주 불가라 기본 배제(config 상향만). (§6-8)
- **이식성(친구 공유)** = 멀티유저 아님(한 인스턴스 다중 사용자 X) — **다중 하드웨어 배포**. 단일 사용자 가정(ADR-002) 유지. 적응은 **config 모델 교체**로 흡수(분기 코드 X). 최소 사양 명시 필요(VRAM 8GB+, CPU-only 불가). 실제 배포 도구(installer/자동 감지/setup 마법사)는 **v4 1차 후순위**. (§6-1 확장)

**미결: 없음.** ADR-021~031 초안 발행 완료(전부 **Proposed**). native 의존 027/030/031 은 **Phase v4-0 Tauri 탐사 go 후 Accepted**. 탐사 go 시 기존 004/010/015 를 `Superseded by`로 갱신. 다음 단계 = **Phase v4-0 Tauri 탐사**(`docs/agent-tasks/v4/phase-0-tauri-spike.md`).

## 1. 핵심 변화 — 단발 응답 코치 → 지속 에이전트

**v3 현재**:
- WS 연결 = 운동 세션. 끊으면 끝.
- 종목 4개 (풀업/푸시업/스쿼트/플랭크) 고정
- 사용자가 앱을 켤 때만 코치가 등장

**v4 지향**:
- 백엔드는 **항상 살아있는 에이전트**
- 사용자가 가끔 말 걸고, 에이전트가 일정/컨디션을 알아서 추적
- 운동 코치 → **나만의 헬스 매니저 / 일과 비서** 까지 확장
- 메모리, 도구 사용, 능동 알림

## 2. 후보 모듈 (각각 ADR 단위로 검토)

### ADR-022 (가칭) Google Calendar 연동
- OAuth2 (Desktop client) → 캘린더 API
- 운동 일정 자동 등록 (사용자가 "수금 아침 푸시업" 정하면 캘린더에 반복 일정)
- 일정 직전 → 알림 + 코치 세션 자동 시작
- 캘린더의 다른 일정 (미팅 등) 도 읽어서 "오늘 저녁 8시까지 일정 없으니 그때 풀업 어때요?" 같은 능동 제안

### ADR-023 (가칭) 컨디션 트래킹
- 매일 체크인 (자가 보고: 피로도/근육통)
- 수면/걸음수 API 연동 (Apple Health, Google Fit) — **연동 우선순위 의문**
- LLM 이 컨디션 보고 운동 강도 조절

### ADR-024 (가칭) 운동 플랜 LLM 에이전트
- "다음 주 풀업 5회 목표" → 백엔드가 주간 플랜 자동 생성
- 진척도 추적 + 실패 시 플랜 조정
- tool-use 패턴: LLM 이 calendar/db/health API 를 도구로 호출

### ADR-025 (가칭) 멀티턴 + 영속 메모리
- v3 는 WS 세션 안에서만 기억
- v4: SQLite + vector DB (예: chromadb) 로 사용자 발화 영속 저장 → RAG
- 사용자가 한 번 말한 선호/제약 (예: "왼쪽 어깨가 안 좋음") 영구 기억

### ADR-026 (가칭) 종목 확장 + 사용자 정의
- 현재 4종 → 확장
- 사용자 정의 운동 추가 가능
- 종목별 cue 모음, 영상 가이드 링크 (옵션)

### ADR-027 (가칭) 능동 알림 / 백그라운드 동작
- v3 는 사용자가 앱 켤 때만 동작
- v4: 백그라운드 스케줄러 → 운동 시간 다가오면 알림
- 데스크탑 알림 시스템 활용 (Windows toast / macOS notification center)
- **PWA 한계 분명 — native 데스크탑 앱 재검토 필요**

### ADR-028 (가칭) 첫 운동일 체력검증 + 대화형 운동 계획 수립 (v3에서 이월)
> 2026-06-08 사용자 요청. v3 에서는 능동 코치가 곧장 운동을 *제안*만 하고(propose_set),
> 신규 사용자의 실제 체력을 모른 채 고정값(예: 푸시업 10회 3세트)을 던진다. v4 에서
> 첫 세션을 **상의형 온보딩 코칭**으로 격상.

- **현재(v3) 한계**: 온보딩 폼(`Onboarding.tsx`)에 체력 측정 입력칸(풀업/푸시업/스쿼트/플랭크 최대)이 있고 `/onboarding` 에 저장되지만, 코치 대화 흐름에서 이를 **능동적으로 끌어내거나 그 결과로 계획을 함께 세우는 대화**는 없음.
- **v4 지향**:
  - 첫 세션 시작 시 코치가 사용자와 **대화로 체력 수준을 검증** (예: "푸시업 몇 개까지 가능하세요?" → 답 → "그럼 그 70%로 시작해볼게요").
  - 검증 결과로 **세트·반복·휴식·종목을 함께 결정**하고 동의를 받아 첫 루틴 확정.
  - 결과를 프로필/플랜에 저장 → 이후 세션·주간 플랜(ADR-024)의 기준선으로 사용.
  - 재평가 트리거: N주 경과 또는 사용자가 "너무 쉬움/어려움" 피드백 시 재측정 대화.
- **연관**: ADR-024(플랜 에이전트)의 입력, ADR-023(컨디션)·ADR-025(영속 메모리)와 결합. 단발 propose_set 을 **대화형 plan-building tool-use** 로 대체하는 게 핵심.
- **범위 메모**: v3 에서는 의도적으로 **보류**(YAGNI + master 동결). 단발 제안으로 충분하다고 판단. v4 에서 대화형 코칭의 첫 적용 사례로 다룬다.

## 3. 기술 결정 재검토 후보

| 영역 | v3 | v4 검토 |
|---|---|---|
| TTS | **faster-qwen3-tts (Qwen3-TTS-12Hz-1.7B-Base + faster 백엔드)** ✅ 통합 완료 ([상세](../_archive/v3/faster-qwen3-tts.md)) | 유지 또는 token-streaming 도입(첫 청크 ~390ms) |
| STT | faster-whisper large-v3-turbo | 유지 또는 더 작은 모델로 latency 개선 |
| LLM | qwen3:8b (Ollama) | tool-use 모델 + function calling 강화 |
| 에이전트 프레임워크 | 자체 구현 (StructuredOllamaProcessor + ConfirmRule) | LangGraph / LlamaIndex / 직접 짠 framework 결정 |
| 음성 파이프라인 | Pipecat | Pipecat 유지 vs 직접 짤지 (Pipecat 1.3 API churn 봤음) |
| UI | React PWA | PWA 한계 — Electron / Tauri 재검토? |
| 데스크탑 알림 | 없음 | OS native 알림 추가 (Tauri 등 후보) |
| 멀티유저 | 단일 사용자 (ADR-002) | 유지 vs 확장 결정 |

## 4. v3 → v4 마이그레이션 전략

1. **v3 master 동결** (현재 상태)
2. v4 새 브랜치 `v4-rewrite` 또는 별도 repo
3. v4 새 ADR 발행 (v3 ADR 과 충돌 가능 — v4 가 우선)
4. v3 작동하는 동안 v4 점진 개발
5. 데이터 마이그레이션: v3 SQLite 스키마 → v4 확장
6. v4 베타 안정화되면 master 교체

## 6. Deep Interview 질문 목록 (다음 세션에서 사용)

다음 세션이 사용자와 한 번에 한 묶음씩 대화하면서 채울 질문들. 답이 채워지면 ADR
초안 작성으로 이동.

### 6-1. 비전과 범위
- v4 의 한 줄 비전은? ("개인 헬스 매니저" 정도?)
- 운동 외 어디까지 확장? (영양 추적? 수면? 일과 비서?)
- 단일 사용자 가정 유지? 가족/친구 공유 같은 기능?

### 6-2. Google Calendar 연동
- OAuth 흐름은 한 번만 + 영구 저장 OK?
- 사용자가 직접 등록한 일정 외에 에이전트가 자동 등록도? 자동 등록 시 사용자 확인 단계 있나?
- 캘린더의 미팅 같은 다른 일정도 읽어서 운동 시간 추천?
- 캘린더가 없는 사용자는?

### 6-3. 능동 알림
- 데스크탑 푸시 알림 필요? (Windows toast 등)
- 알림 시간 어떻게 (운동 시간 30분 전? 사용자 설정?)
- 음소거 시간대 (밤?)
- 알림에서 바로 코치 세션 시작 가능해야 하나?

### 6-4. 컨디션/건강 데이터
- Apple Health / Google Fit 연동 우선순위는?
- 일일 체크인 강제 vs 선택?
- 컨디션 데이터로 운동 강도 자동 조절 vs 사용자 확인?

### 6-5. 운동 플랜
- "이번 주 X회 목표" 같은 목표 기반 vs 일일 자유?
- 실패하면 자동 조정 vs 그대로 유지?
- 주간 / 월간 리뷰 자동 생성?

### 6-6. 메모리 / 학습
- 사용자 선호 (예: "어떤 운동 싫음") 어디까지 기억?
- 부상 / 제약 사항 영구 기억 (예: "왼쪽 어깨")?
- vector DB 도입 vs SQLite 만으로?

### 6-7. 종목 / 운동 데이터
- 종목 확장 우선순위? (덤벨 운동? 유산소?)
- 영상 가이드 / 사진 링크 필요?
- 사용자 정의 종목 우선순위?

### 6-8. 기술 스택 결정
- Pipecat 유지 vs 자체 음성 파이프라인 (1.3 의 API churn 봤음)
- LangGraph / LlamaIndex 같은 에이전트 framework 도입?
- Electron / Tauri 로 native 알림 + 백그라운드 동작?
- LLM 모델 변경 (qwen3:8b 외)?

### 6-9. 데이터 / 프라이버시
- 로컬 only 유지 (v3 ADR-002)?
- 캘린더 연동은 외부 API — 어디까지 허용?
- 음성 데이터 영구 저장 vs 휘발?

### 6-10. UX 변화
- v3 의 4-모드 (C2C/C2S/S2S/S2C) 유지?
- 항상 음성 켜져있는 "ambient" 모드?
- 운동 중 멀티태스킹 (음악 등) 어디까지?

### 6-11. 첫 운동일 체력검증 + 대화형 계획 수립 (ADR-028)
- 첫 세션을 대화형 체력검증으로 시작? (코치가 질문 → 사용자 답 → 계획 제안)
- 검증 방식: 자가 보고(몇 개 가능?) vs 실측 세트(1세트 최대치 측정)?
- 검증 결과로 시작 강도를 어떻게 도출? (예: 최대치의 X%)
- 재평가 주기/트리거 (N주? "너무 쉬움" 피드백?)
- 온보딩 폼의 기존 측정 입력과 대화형 검증의 관계 (폼 생략 가능? 폼 우선?)
