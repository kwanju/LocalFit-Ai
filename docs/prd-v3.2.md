# PART 1 — CORE PRD

## 1. 제품 개요

LocalFit AI는 로컬 LLM으로 동작하는 개인 AI 피트니스 코치입니다. 사용자의 환경(집·헬스장·새벽 등)에 맞춰 입력과 출력 모드를 자유롭게 조합할 수 있고, AI 코치가 운동 카운팅, 대화 기반 일정 조정, 안전 모니터링을 수행합니다.

### 1-1. 제품 한 줄 정의

> *"화면을 안 봐도 운동할 수 있고, 말을 못 해도 코칭받을 수 있는 오프라인 AI 트레이너."*

### 1-2. 핵심 가정 (v3.2 갱신)

- 로컬 LLM (Ollama + Qwen 3.5 9B / Gemma 4 E4B 듀얼)으로 충분한 코칭 품질이 나온다 `[ADR-003, 004]`
- faster-whisper(large-v3-turbo) STT + Kokoro/Qwen3-TTS 교체형으로 한국어 S2S 파이프라인 2~3초 내 응답 가능하다 `[ADR-005, 006]`
- 사용자 본인 1인 사용 (가족 공유 등은 후순위, 멀티유저 코드 미작성) `[ADR-002]`
- 부상 발화는 LLM 응답 외에 규칙 기반으로 즉시 처리한다 (Safety Guard 인터셉터)
- RTX 5090(32GB) + 64GB RAM 환경에서 모델 동시 상주 가능 `[ADR-011]`

### 1-3. MVP 범위 및 배포 전략 (v3.2 신규)

아키텍처 설계 단계에서 확정된 MVP 경계와 확장 로드맵입니다.

| 구분 | MVP (P0) | 확장 (P1+) |
|---|---|---|
| 토폴로지 | 데스크탑 단독, 백엔드 `127.0.0.1` 바인딩 | 폰 PWA + Tailscale + Wake-on-LAN |
| UI 접근 | 데스크탑 브라우저 | 폰 브라우저 동시 (헬스장 LTE) |
| 환경 | Windows 네이티브 + NVIDIA, uv venv | (동일) |
| 모델 배포 | 첫 실행 시 셋업 스크립트로 다운로드 | PWA 셋업 화면 |
| 사용자 | 본인 1인 | GitHub 공개 시 각자 자기 인스턴스 |

> **참고**: "오프라인 우선"과 "헬스장 사용"은 부분 충돌하므로, MVP는 데스크탑 단독으로 시작하고 폰·헬스장 시나리오는 P1으로 분리했습니다. 상세는 ADR-002 참조.

## 2. 입출력 모드 매트릭스 ★ 핵심

사용자의 환경에 따라 입력 채널(음성·채팅)과 출력 채널(음성·채팅)을 자유롭게 조합합니다. 입력과 출력이 독립적이라는 게 핵심 — 환경에 맞춰 한쪽만 바꿀 수 있어야 합니다. 구현상 STT/TTS 어댑터의 on/off 조합이 곧 모드입니다 `[ADR-010]`.

### 2-1. 4가지 모드

| 모드 | 입력 → 출력 | 주요 사용 환경 | 활용 예시 |
|---|---|---|---|
| S2S | 음성 → 음성 | 집·홈짐 | 핸즈프리. 화면 없이 코치와 자유 대화 |
| C2S ★ | 채팅 → 음성 | 헬스장 | 폰 입력, 이어폰 코칭 — 말하기 어려운 환경 |
| C2C | 채팅 → 채팅 | 새벽·도서관·심야 | 소리 안 나게. 완전 무음 환경 |
| S2C | 음성 → 채팅 | 조용히 답만 보고 싶을 때 | 드물게 쓰임. 사용자 선택 옵션 |

### 2-2. 모드 전환 정책

- 기본 모드: 사용자가 환경 설정에서 지정 (집·헬스장 프리셋)
- 자동 전환 없음 — 사용자가 명시적으로 변경. 의도치 않은 전환이 더 짜증나는 경험
- 세션 중 전환 가능: 채팅창 상단 아이콘으로 1탭 변경
- 이어폰 연결 감지 시: "이어폰 연결됐어요. 음성 출력으로 바꿀까요?" 알림만 띄움 (자동 X)

### 2-3. C2S(헬스장 모드) 특화 설계

헬스장 사용 시 가장 빈번한 시나리오. UI/UX를 이 모드 기준으로 최적화. (P1에서 폰 PWA로 완성)

| 요소 | C2S 모드 설계 |
|---|---|
| 채팅 입력 | 원터치 빠른 응답 버튼 우선 ("세트 완료", "한 세트 더", "휴식 30초 추가") |
| AI 카운팅 | 화면 큰 숫자 + 이어폰 음성 카운팅 동시 출력 |
| 인터럽트 | 화면 탭 한 번으로 코치 음성 즉시 중단 |
| 휴식 알림 | 음성 + 화면 진동. 헤드폰 빠질 가능성 대비 |
| 폼 기록 | 음성 폼 코칭 대신 "이번 세트 어땠어요?" 채팅 4지선다 제공 |

### 2-4. 모드별 기능 가용성

| 기능 | S2S | C2S | C2C | S2C |
|---|---|---|---|---|
| AI 음성 카운팅 | ✓ | ✓ | 화면 대체 | ✓ |
| 코칭 대화 | ✓ | ✓ | ✓ | ✓ |
| 인터럽트 | 음성 | 화면 탭 | 화면 탭 | 음성 |
| 휴식 멘트 | 음성 | 음성 | 채팅 | 채팅 |
| 부상 감지 | 음성 키워드 | 채팅+버튼 | 채팅+버튼 | 음성 키워드 |

## 4. 아키텍처 개요

> 상세 컴포넌트 설계·다이어그램·인터페이스는 부록 H의 ADR 및 `docs/architecture/`를 참조합니다. 본 절은 요약만 유지합니다.

### 4-1. 모드별 파이프라인

| 모드 | 파이프라인 | 총 지연 (목표) |
|---|---|---|
| S2S | VAD → STT → 의도 분류 → LLM → TTS | 2~3초 |
| C2S | 채팅 입력 → 의도 분류 → LLM → TTS | 1~2초 |
| C2C | 채팅 입력 → 의도 분류 → LLM → 채팅 출력 | 0.8~1.5초 |
| S2C | VAD → STT → 의도 분류 → LLM → 채팅 출력 | 1.5~2.5초 |

> **핵심 설계 원칙**
> - STT와 TTS는 독립 어댑터. 각자 켜고 끌 수 있어야 4-모드 매트릭스가 동작.
> - 의도 분류 + LLM 응답은 모든 모드 공통. 입출력만 갈아끼움.
> - 카운팅 엔진은 LLM과 분리. LLM 지연 중에도 카운팅은 끊김 없이 계속.

### 4-2. 기술 스택 (v3.2 확정)

| 레이어 | 확정 기술 | ADR |
|---|---|---|
| 메인 언어 | Python 3.11+ | 001 |
| LLM 런타임 | Ollama | 003 |
| LLM 모델 | Qwen 3.5 9B + Gemma 4 E4B (config 전환) | 004 |
| STT | faster-whisper large-v3-turbo (한국어) | 005 |
| TTS | Kokoro + Qwen3-TTS (교체형 어댑터) | 006 |
| VAD | silero-vad | 012 |
| DB | SQLite + SQLModel | 007 |
| 백엔드 | FastAPI + WebSocket | 008 |
| UI | React + Vite + TypeScript + Tailwind (PWA) | 009 |
| 패키지 매니저 | uv (Python) + pnpm (JS) | 015 |
| 환경 | Windows 네이티브 + uv venv (Docker/WSL2 미사용) | 016 |

최소 사양: RAM 8GB / 권장: RAM 16GB + GPU. 개발·운영 기준 환경: Windows + RTX 5090(32GB) + 64GB RAM. 모델 동시 상주 전략은 ADR-011 참조.

## 5. AI 카운팅

### 5-1. 운동별 모드

| 운동 | 모드 | 박자 구성 | 음성 예시 (C2C는 화면 표시) |
|---|---|---|---|
| 풀업 | 메트로놈 | 올라가기 + 내려가기 | "하나 — 올라가요! / 내려오세요" |
| 푸시업 | 메트로놈 | 내려가기 + 올라가기 | "둘 — 내려가요! / 올라와요!" |
| 스쿼트 | 메트로놈 | 앉기 + 서기 | "셋 — 앉아요! / 올라와요!" |
| 플랭크 | 타이머 | 초 단위 카운트다운 | "30초, 코어에 힘! / 15초 남았어요" |

### 5-2. 카운팅 설정값

| 파라미터 | 기본 | 범위 |
|---|---|---|
| 박자 간격 (메트로놈) | 2초 | 1~4초 |
| 플랭크 목표 시간 | 30초 | 10~120초 |
| 세트 간 휴식 | 60초 | 15~120초 |

> 구현 주의: 박자 스케줄러는 `time.monotonic()` 기반 절대 시각 사용 (sleep 누적 금지). 박자 정확도 ±10% (코딩 컨벤션 참조).

## 6. 안전 정책 (요약)

개인 프로젝트라도 안전과 면책은 최소 수준으로라도 명시. 상세는 부록 B 참조. 구현은 Safety Guard 인터셉터로 모든 입력을 LLM 이전에 가로챔.

> **필수 3가지**
> 1. 의료 면책 고지 — 최초 실행 시 1회 표시. "의료 조언 아님" 명시
> 2. 부상 키워드 감지 — "아파", "다쳤어" 발화/입력 시 즉시 운동 중단 + 전문의 상담 안내
> 3. LLM 안전 프롬프트 하드코딩 — 통증 무시·운동 강행 권유 금지 규칙을 시스템 프롬프트 최상단에 고정

### 6-1. 의료 면책 고지 (필수 문구)

> **LocalFit AI는 피트니스 가이던스 앱이며, 의료 기기나 의료 전문가를 대체하지 않습니다. 통증, 부상, 또는 건강 이상이 있다면 운동을 중단하고 의료 전문가와 상담하세요.**

> **구현 노트 (2026-05-29, 단일 사용자 MVP 결정):** 위 "필수 3가지"의 1번(면책 고지 "최초 실행 시 1회 표시")은 **적용하지 않는다.** 혼자 쓰는 앱(ADR-002 단일 사용자)이라 법적 면책의 실익이 낮다고 판단하여, 첫 실행 모달 없이 UI 설정 화면 맨 아래 접이식(옵션) 항목으로만 노출한다. 부상 키워드 인터셉터(2번)와 LLM 안전 프롬프트(3번)는 그대로 유지한다. 멀티유저/외부 배포 전환 시 본 결정을 재검토할 것.

## 7. MVP 출시 기준 & 미결 사항

### 7-1. MVP 동작 확인 체크리스트

이 5가지가 안정적으로 동작하면 v1 출시. KPI보다 "이게 되면 끝" 기준.

- [ ] 4가지 모드(S2S, C2S, C2C, S2C) 모두 끊김 없이 동작
- [ ] AI 카운팅이 메트로놈/타이머 자동 전환, 박자 정확도 ±10%
- [ ] 코칭 대화에서 일정 변경 시나리오 5가지가 자연스럽게 처리됨
- [ ] 부상 키워드 감지 시 즉시 중단 + 면책 고지 100% 동작
- [ ] LLM 응답 지연 4초 초과 시 폴백 동작, 카운팅은 끊기지 않음

### 7-2. 미결 사항 (v3.2 갱신)

| # | 질문 | 상태 (v3.2) |
|---|---|---|
| 1 | 한국어 TTS 모델 | **결정됨**: Kokoro + Qwen3-TTS 교체형. 청취 비교는 P0 첫 주 `[ADR-006]` |
| 2 | C2S 빠른 응답 버튼 목록 (10~15개) | 진행 중: P0 UI 작업(Phase 5C)에서 확정 |
| 3 | UI: Electron vs PWA | **결정됨**: React + Vite PWA `[ADR-009]` |
| 4 | 부상 키워드 전체 목록 (최소 20개) | 진행 중: P0 Safety Guard 작업(Phase 3A)에 포함 |
| 5 | 오프라인 모드 시 동기화 정책 | P1 유지 (폰 도입 시 결정) |

# PART 2 — 부록 (Modular Appendices)

각 부록은 독립 모듈로, 본문 변경 없이 추가·제거 가능합니다. MVP 단계에서는 부록 A·B·H를 활용하고, 나머지는 필요해질 때 채워 넣습니다.

> **부록 목록**
> A. 사용자 온보딩 상세 — MVP 필요
> B. 안전 정책 상세 — MVP 필요 (요약은 본문 6장)
> C. 코치 페르소나 (P2) — MVP는 1종만

## 부록 A. 사용자 온보딩 상세

최초 1회 실행되는 온보딩 플로우. 강도 조정의 기준점을 만들기 위함.

### A-1. 3단계 압축 플로우

| 단계 | 내용 | 소요 |
|---|---|---|
| 1 | 목표 설정 + 운동 가능 요일·시간대 (음성 또는 채팅) | ~2분 |
| 2 | 체력 측정 (테스트 진행) or 자가 평가 선택 | ~3~10분 |
| 3 | 첫 루틴 자동 생성 + 확인 | ~1분 |

### A-2. 체력 측정 방식

| 운동 | 측정 | 결과 활용 |
|---|---|---|
| 풀업 | 최대 렙 1세트 (안 되면 네거티브) | 초기 렙 = 최대 × 0.6 |
| 푸시업 | 최대 렙 1세트 | 초기 렙 = 최대 × 0.65 |
| 스쿼트 | 최대 렙 1세트 | 초기 렙 = 최대 × 0.65 |
| 플랭크 | 최대 유지 시간 | 초기 시간 = 최대 × 0.7 |

### A-3. 자가 평가 기본값

- 초급: 풀업 3×3 / 푸시업 3×8 / 스쿼트 3×10 / 플랭크 3×20s
- 중급: 풀업 3×6 / 푸시업 3×12 / 스쿼트 3×15 / 플랭크 3×30s
- 고급: 풀업 3×10 / 푸시업 3×20 / 스쿼트 3×20 / 플랭크 3×60s

## 부록 B. 안전 정책 상세

### B-1. 부상·통증 발화 4단계 대응

| 키워드 예시 | 위험 | 즉각 대응 | 후속 조치 |
|---|---|---|---|
| "뻐근해", "피곤해" | 낮음 | "잠깐 쉬고 계속할까요?" | 강도 일시 낮춤 |
| "무릎 아파", "어깨 쑤셔" | 중간 | 세션 일시정지. "어디가 아픈가요?" | 해당 부위 제외 or 종료 |
| "찢어지는 느낌" | 높음 | 즉시 전체 중단 + 면책 고지 | 세션 강제 종료 + 전문의 안내 |
| "숨 안 쉬어져" | 응급 | "즉시 멈추고 119에 연락" | 앱 강제 종료 권고 |

### B-2. LLM 안전 프롬프트 하드코딩

- 시스템 프롬프트 최상단에 안전 규칙 삽입. 사용자 요청으로 override 불가
- 규칙: "사용자가 통증·부상을 언급하면 즉시 운동 중단 권고 + 전문의 상담 안내. 통증 무시 또는 운동 강행 권유 금지"
- 응급 키워드는 LLM 우회, Safety Guard 규칙 기반으로 즉시 처리

### B-3. 과훈련 감지

| 조건 | 임계값 | 대응 |
|---|---|---|
| 주간 볼륨 급증 | +20% 이상 | 휴식 권유 멘트 |
| 연속 고강도 3일↑ | 90%+ 강도 | 경량 세션 제안 |
| 컨디션 하락 추세 | 3회 연속 5점↓ | 주간 볼륨 자동 하향 |

### B-4. 점진적 증량 원칙

- 주간 볼륨 증가: 최대 +10%. LLM이 이 범위 초과 제안 불가
- 렙 수: 한 번에 +2렙 / 세트 수: +1세트 한도
- 부상 복귀: 50% → 70% → 90% → 정상 (4주 점진)

## 부록 C. 코치 페르소나 (P2 고려)

MVP는 "친근한 트레이너" 1종만. P2에서 다중 페르소나 추가 검토. 페르소나별 음색은 TTS 교체형 구조(ADR-006)를 동시 다중으로 확장하여 구현 가능.

| 페르소나 | 스타일 | TTS 톤 | 프롬프트 방향 |
|---|---|---|---|
| 친근한 트레이너 (MVP) | 부드럽·공감·동기 | 보통·따뜻 | 질문·공감·칭찬 |
| 군인 코치 (P2) | 엄격·박력·간결 | 빠름·강함 | 명령형·약점 지적 |
| 과학 분석가 (P2) | 논리·정확 | 차분·중립 | 수치·이유 설명 |

## 부록 D. QA 시나리오 (출시 전 채움)

MVP 체크리스트(본문 7-1)와 별개. 출시 직전 단계(Phase 6)에서 채워 넣을 상세 테스트 목록.

**음성 처리**
- 소음 환경 STT 정확도 (집·헬스장·카페 시뮬레이션)
- 인터럽트 타이밍 엣지 케이스
- 매우 짧은 발화("네", "아") 감지 vs 소음 구분

**모드 전환**
- 세션 중 S2S → C2S 즉시 전환 동작
- 이어폰 연결/해제 감지 알림
- 화면 잠금 상태에서 C2S 동작 확인 (Screen Wake Lock)

**안전**
- 부상 키워드 20개 변형 표현 테스트
- "부상 무시하고 계속해" 시도 → LLM 거부 확인

**TTS 청취 (v3.2 신규)**
- Kokoro vs Qwen3-TTS 한국어 30문장 청취 → `config.tts.active` 확정

**오프라인/에러**
- 네트워크 없이 모든 P0 기능 동작
- LLM 강제 지연 4초+ 시 폴백
- 앱 강제 종료 후 세션 복원

## 부록 E. 워밍업·쿨다운 (운영 콘텐츠)

운동 동작 명세는 PRD가 아닌 운영 콘텐츠. 개발 단계에서 운동 DB로 채움. 본 부록은 참고용 가이드.

### E-1. 워밍업 (세션 전 3~5분)

- 제자리 걷기 / 가벼운 조깅 (1분)
- 팔 돌리기·어깨 회전 (30초)
- 고관절 돌리기·레그 스윙 (30초)
- 가벼운 스쿼트 30% 강도 (1분)
- 주 운동 특화 동적 스트레칭 (1분)

### E-2. 쿨다운 (세션 후 3~5분)

- 제자리 걷기 + 심박수 안정화 (1분)
- 사용 근육 정적 스트레칭 (2~3분, 각 20~30초)
- 호흡 정리 + 마무리 멘트 (1분)

### E-3. 건너뛰기 정책

- 첫 요청: "2분으로 줄여서 할까요?" 축약 제안
- 두 번째 요청: 강요 없이 진행. 기록만 저장 (장기 패턴 분석용)

## 부록 F. 영양·수분 리마인더 (P2 검토)

MVP 범위 밖. 제품의 핵심은 운동 코칭이며 영양 영역 확장은 의도적으로 후순위.

- 세션 전후 수분 섭취 알림
- 운동 후 30분 단백질 타이밍 안내
- 필수 면책 고지: "일반 운동 가이드라인이며 의료·영양 조언 아님"
- 확장 시 별도 영양사 자문 권장

## 부록 G. 장기 패턴 인식 (P1)

운동 기록 데이터가 충분히 누적된 후(최소 4주) 활성화. MVP 단계에서는 비활성. `InteractionLog`·`SetLog`·`ConditionLog` 테이블이 분석 기반.

- 월간 요일별 컨디션·완료율 패턴 감지
- 정체기 자동 인식 (3주 같은 무게·렙)
- 주간 단위 코치 선제 제안 ("목요일마다 피곤하셨으니 목요일은 가볍게 할까요?")
